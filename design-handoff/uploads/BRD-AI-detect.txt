ДОКУМЕНТ БІЗНЕС-ВИМОГ (BRD)Проект: SaaS-платформа «Content Optimizer & AI Humanizer»1. ПАСПОРТ ПРОЕКТУ ТА ЗАГАЛЬНІ ВІДОМОСТІНазва продукту: AI Humanizer / Content Optimizer (робоча назва).Тип продукту: SaaS-платформа (Software as a Service) за моделлю підписки з лімітами на обсяг слів (кредити).  Бізнес-модель: Продаж місячних/річних пакетів слів (наприклад, 15 000 слів за базовим тарифом, 50 000 слів за просунутим тарифом).  Цільова аудиторія (ЦА): SEO-маркетологи, копірайтери, маркетингові агенції, контент-мейкери та власники сайтів, які масштабують генерацію контенту за допомогою ШІ, але стикаються з ризиком втрати контрактів або падіння у видачі через автодетектори.  2. БІЗНЕС-ЦІЛІ ТА МЕТРИКИ УСПІХУФінансова ціль: Досягнення показника регулярного щомісячного доходу (MRR) у розмірі 15 000 доларів США протягом перших 6–9 місяців з моменту запуску MVP.  Технічна валідація: Зниження ймовірності детекції AI-контенту до рівня менше 10–15% на основних детекторах (GPTZero, Turnitin, Copyleaks, Originality.ai).  Збереження цінності контенту: Забезпечення читабельності оптимізованого тексту на рівні не нижче 60 балів за шкалою Flesch Reading Ease.  Утримання клієнтів (Retention): Зниження рівня відтоку клієнтів (Churn) нижче 5% на місяць шляхом забезпечення високої якості тексту (уникнення генерації хаотичного «словникового супу» під час рерайтингу).  3. ФУНКЦІОНАЛЬНІ ВИМОГИ (SYSTEM FUNCTIONAL REQUIREMENTS)3.1. Ядро системи: Чотиритрирівневий конвеєр «олюднення» (4-Layer Humanization Pipeline)Система повинна обробляти текст через послідовні технологічні шари для повного руйнування статистичного сліду ШІ :  Layer 1 (LLM Rewrite): Переписування за допомогою комерційних API (Claude 3.5, Llama 3 через OpenRouter).  Вимога: Налаштування системних промптів для примусової ін'єкції перплексії (випадковості вибору слів) та сплесковості (чергування довжини речень).  Конфігурація: Впровадження 13 стилістичних тонів (Conversational, Persuasive тощо) та 5–6 стильових профілів (Academic, Professional, Creative).  Layer 2 (Non-LLM Post-Processing): Безсерверна скриптова фільтрація.  Вимога: Автоматичне виявлення та заміна за базою регулярних виразів (Regex) маркерів «AI-слапу» (слова delve, tapestry, crucial, vibrant, nestled, underscore, furthermore, in conclusion). Виконання синонімічних замін (понад 500 варіантів).  Layer 3 (Multi-Model Chain - Опціонально): Перехресне змішування.  Вимога: Можливість послідовного каскадного пропускання тексту через різні архітектури LLM для нейтралізації специфічного «почерку» одного провайдера.  Layer 4 (Final Polish & Readability Guard): Фільтр якості.  Вимога: Валідація тексту на збереження семантичної цілісності (запобігання логічним помилкам та втраті закликів до дії — CTA).  3.2. Алгоритм внутрішнього циклу детекції (Internal Detection Loop)Щоб уникнути обмежень повної генерації на довгих текстах (більшість гуманізаторів некоректно працюють з текстами обсягом понад 1500 слів ), система має містити наступну логіку :  Сегментація (Chunking): Автоматичне розбиття введеного тексту на фрагменти обсягом $\le$ 2500 слів за межами речень.Сканування (Sentence-level Scoring): Надсилання фрагментів на оцінку у внутрішній або зовнішній евристичний детектор.Ітеративне доопрацювання (Auto-Refinement): Якщо речення отримує оцінку штучності вище 15–20%, система автоматично витягує саме ці речення та повторно відправляє їх на агресивну обробку (режими «Aggressive» або «Ninja»), зберігаючи вже очищені речення незмінними.3.3. Модуль метрик та аналізу читабельності (Readability Engine)Візуалізація якості тексту для SEO-копірайтерів у реальному часі на основі наступних метрик :Flesch Reading Ease (FRE): Оцінка легкості сприйняття тексту за формулою:$$FRE=206.835-(1.015\times{ASL})-(84.6\times{ASW})$$
(де $ASL$ — середня довжина речення в словах, $ASW$ — середня кількість складів у слові).Flesch-Kincaid Grade Level (FKGL): Оцінка складності за рівнем освіти США за формулою:$$FKGL=0.39\times{ASL}+11.8\times{ASW}-15.59$$.Технічна вимога: Використання надійних npm-бібліотек text-readability-ts або flesch-kincaid для розрахунку параметрів на рівні сервера.3.4. Вимоги до Інтерфейсу користувача (UI/UX)Головний екран редактора: Розділений на два функціональні блоки (ліва панель — оригінальний текст; права панель — оптимізований людський текст).Інструменти керування:Селектор режимів інтенсивності обробки (Light, Medium, Aggressive, Ninja).Випадаючі списки для вибору 13 пресетів тону та 5 стилів.Інтерактивна підсвітка тексту: Візуальне маркування речень різними кольорами (червоний — висока ймовірність ШІ, жовтий — змішаний, зелений — пройдено детекцію).Функція "Альтернативи": Клік на будь-яке речення відкриває спливаюче вікно з 3 альтернативними варіантами перефразування, згенерованими при температурі моделі $1.0$.3.5. Білінг, Кредитна система та Stripe Sync EngineДля безперебійного масштабування бізнесу без технічних збоїв у нарахуванні статусів :Вимога: Використання інтеграції Stripe Sync Engine через панель Supabase для автоматичної дзеркальної синхронізації клієнтів, транзакцій та підписок з платіжного шлюзу до PostgreSQL (схема stripe).База даних (Схема public):Таблиця public.prices з полем credits (визначає кількість доступних слів для кожного тарифу).Автоматичний тригер PostgreSQL, який при успішній оплаті підписки (статус active) або інвойсу нараховує відповідну кількість кредитів користувачу.Логіка списання: API-запит на обробку тексту повинен спершу виконувати перевірку балансу користувача (public.users_balance) з увімкненою політикою безпеки рядків (Row Level Security — RLS), і лише після успішної транзакції списувати точну кількість символів/слів.4. НЕФУНКЦІОНАЛЬНІ ВИМОГИ (NON-FUNCTIONAL REQUIREMENTS)4.1. Безпека та конфіденційність данихPrivacy-First політика: Тексти користувачів не зберігаються на сервері на постійній основі для навчання сторонніх моделей. Вони кешуються виключно в межах сесії користувача або локального сховища (localStorage) у браузері.Безпека API-ключів: Усі зовнішні API-запити до LLM-провайдерів (OpenRouter, DeepInfra) та детекторів (GPTZero) повинні проксуватися через захищені серверні маршрути (Next.js Server Actions або Supabase Edge Functions), виключаючи витік приватних ключів у клієнтську частину додатку.4.2. Швидкість та лімітиШвидкість обробки: Середній час олюднення та аналізу тексту не повинен перевищувати 20 секунд на 1000 слів.Стабільність лімітів: Текст обробляється порційно, щоб уникнути помилок тайм-ауту API або перевищення контекстного вікна моделей.5. ТЕХНОЛОГІЧНИЙ СТЕК ДЛЯ VIBECODING В CURSORДля досягнення максимальної швидкості розробки ШІ-редактором Cursor, технологічний стек має складатися з максимально популярних та задокументованих інструментів :Фронтенд: Next.js (App Router) на базі React Server Components (RSC) та TypeScript.Стилізація та UI: Tailwind CSS у комбінації з бібліотекою компонентів Shadcn UI / SaaS UI.Бекенд/База даних: Supabase (PostgreSQL) із налаштованими політиками RLS.Платежі: Stripe + Stripe Sync Engine (Supabase Integration).Інтеграції LLM: OpenRouter API (дозволяє керувати моделями Claude, Llama, Mistral через один клієнт).Інтеграція детектора: GPTZero API (запити POST https://api.gptzero.me/v2/predict/text).6. СТРАТЕГІЯ ЗАПУСКУ ТА ЗРОСТАННЯ (GTM REQUIREMENTS)Продукт повинен містити вбудовані елементи воронки залучення користувачів без прямого рекламного бюджету :Free-to-Paid воронка (Engineering as Marketing):Реалізувати окремий безкоштовний субдомен/цільову сторінку з простим AI-детектором. Користувачі безкоштовно перевіряють тексти на наявність ШІ, і при отриманні високого результату детекції система пропонує цільову дію (CTA): "Олюднити цей текст за один клік" зі спрямуванням у платну версію.Програмне SEO (Programmatic SEO):Генерація тисяч посадкових сторінок під довгі ключові запити (наприклад: "how to bypass Turnitin for essays", "how to make ChatGPT text undetectable") з автоматичною індексацією через протокол IndexNow.Зворотний зв'язок на ранній стадії:Впровадження автоматичного тригера: надсилання емейлу користувачу від імені засновника одразу після реєстрації з простим запитанням: "Яку саме проблему ви намагаєтесь вирішити за допомогою нашого інструменту?" для точного налаштування цільового меседжу.
